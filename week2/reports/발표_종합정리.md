## 1. 프로젝트 한 줄 요약

> **전쟁·정치 불안(GPR) + 시장 공포(VIX)를 입력으로, 원유 가격이 내일 ±2% 이상 크게 움직일 확률을 AI가 예측하는 시스템**

- **타겟**: 원유 WTI (1차). 추후 금·은·천연가스로 확장
- **기간**: 2020-01-02 ~ 2026-03-16 (약 1,558 거래일)
- **과제 구조**: 분류(급변동 감지/방향 예측) + 회귀(수익률 예측)

## 2. 사용 기술 스택

### 데이터 수집
| 도구 | 용도 |
|---|---|
| **yfinance** | Yahoo Finance에서 WTI, 금, 은, 천연가스, VIX 자동 수집 |
| **pandas** | 엑셀/시계열 데이터 처리 |
| **GPR Index (Caldara & Iacoviello)** | FRB 공개 엑셀 — 전쟁·테러 관련 뉴스 기사 빈도로 지정학적 리스크를 수치화한 지표 |

### 데이터 전처리
| 도구 | 용도 |
|---|---|
| **pct_change** | 일간 수익률 계산 |
| **StandardScaler** | 평균 0, 표준편차 1로 정규화 (Z = (X-μ)/σ) |
| **TimeSeriesSplit (5-Fold)** | 시간 순서를 보장하는 교차검증 — 미래 데이터가 학습에 새는 문제(Data Leakage) 방지 |

> **정규화 포인트**: SVM/KNN/LogisticRegression은 스케일에 민감 → StandardScaler 필수. Random Forest/XGBoost는 트리 기반이라 스케일 무관하지만, 비교 공정성을 위해 동일하게 적용

### 머신러닝 모델
| 라이브러리 | 모델 | 역할 |
|---|---|---|
| **scikit-learn** | RandomForest, LogisticRegression, SVM(RBF), KNN | 2주차 비교 대상 |
| **xgboost** | XGBClassifier, XGBRegressor | 최종 선정 모델 |

### 서비스·배포
| 도구 | 용도 |
|---|---|
| **Streamlit** | 데모앱 UI (Toss 스타일) |
| **Python 3.9 + Colab/Local** | 학습 환경 |
| **matplotlib** | 시각화 (ROC, 혼동행렬, Fold별 F1 등) |

### 핵심 알고리즘 설명
**RBF 커널 (SVM에서 사용)**
- 수식: `K(x,y) = exp(-γ||x-y||²)`
- 의미: 데이터를 고차원으로 매핑해 선형 분리 불가능한 문제를 곡선 경계로 분리
- 프로젝트 연관: VIX·GPR의 비선형 관계를 잡을 수 있다는 가정으로 채택

**StandardScaler**
- 수식: `Z = (X-μ)/σ`
- 필요성: VIX(10~80), GPR(0~500), 수익률(-10~+10) 스케일 편차 커서 → 보정 없으면 큰 값 Feature에 모델이 끌림

## 3. 왜 XGBoost인가

### 한 줄 답
> **불균형 데이터에 강하고, 이전 실수에 집중 학습하는 부스팅 구조라 우리 목표(Recall 개선)에 가장 부합하기 때문**

### XGBoost의 동작 원리 (30초 설명)
1. 약한 결정 트리를 **순차적으로** 여러 개 만듦
2. 각 새로운 트리는 **이전 트리가 틀린 샘플에 집중**해서 학습
3. 최종 예측 = 모든 트리의 가중합
4. 정규화(L1/L2)로 과적합 방지, 내부적으로 병렬 처리로 빠름

> Random Forest가 트리를 **독립적·병렬**로 만드는 반면, XGBoost는 트리를 **순차적·보완적**으로 만듦. "놓친 이상변동에 재도전"하는 구조 → **Recall 개선에 구조적으로 유리**

### 5개 모델 비교 결과 (2주차, 5-Fold 평균)
| 모델 | Accuracy | Precision | **Recall** | **F1** |
|------|---------|-----------|------------|--------|
| Random Forest | 0.611 | 0.373 | 0.224 | 0.261 |
| Logistic Regression | 0.558 | 0.340 | 0.260 | 0.277 |
| SVM (RBF) | **0.631** | 0.385 | 0.207 | 0.254 |
| **XGBoost** | 0.590 | 0.365 | **0.286** | **0.324** |
| KNN | 0.575 | 0.335 | 0.179 | 0.218 |

### XGBoost 선정 4가지 근거
1. **Recall 1위 (0.286)** — 우리 목표 지표에서 최고
2. **F1 1위 (0.324)** — 균형 지표에서도 최고
3. **`scale_pos_weight` 내장** — 클래스 불균형(정상 64.5% vs 이상 35.5%) 직접 처리
4. **Classifier + Regressor 통합** — 분류(급변동)·회귀(수익률) 같은 계열로 일관성

### 탈락 모델 한 줄 사유
- **SVM(RBF)**: Accuracy는 1위이나 Recall 꼴찌에 가까움(0.207), 대용량·실시간 배포에 느림
- **KNN**: 모든 지표 하위권, 안정성 부족
- **Random Forest**: 1주차 베이스 대비 크게 못 넘어섰고 Recall 낮음
- **Logistic Regression**: 선형 모델 한계, 비선형 리스크-가격 관계 포착 제한

### XGBoost 하이퍼파라미터 (현재)
```
n_estimators=100     # 트리 100개
max_depth=6          # 트리 최대 깊이
learning_rate=0.1    # 학습률 (기본값)
scale_pos_weight=1.8 # 이상변동 클래스 가중치 (불균형 보정)
eval_metric='logloss'
random_state=42
```
> 주의: **GridSearch 튜닝 안 함**. 3주차에서 최적화 예정 → 교수님 질문 포인트

## 4. 왜 위험경보 임계값 30%인가

### 한 줄 답
> **FN(놓침)이 FP(오경보)보다 치명적인 도메인 + 손실회피 심리 + 현재 기본값(0.5)에서 Recall 28.3%의 약점 → 임계값을 낮춰 놓침을 줄임**

### 논리 3단 구성

**① 임계값이란?**
모델은 "이상변동일 확률(0~1)"을 출력. 이 확률을 "위험/안전"으로 변환하는 기준선이 임계값.
- **낮게** 잡으면 → 조금만 의심돼도 경보 → Recall↑ / Precision↓
- **높게** 잡으면 → 확실할 때만 경보 → Precision↑ / Recall↓

**② 왜 Recall을 우선시하나? (손실회피 이론)**
- **제1종 오류 (FP, 오경보)**: 경보 울렸는데 멀쩡 → 투자자 불편, 금전 손해 없음
- **제2종 오류 (FN, 놓침)**: 경보 안 울렸는데 급변동 → **실제 손실 발생**
- 행동경제학 손실회피 효과: 투자자는 같은 금액이라도 **손실을 이익보다 약 2배 더 크게** 느낌
- 결론: FN 비용 >> FP 비용 → **Recall 우선**

**③ 왜 하필 30%인가?**
- 기본값 0.5에서 Recall 28.3% (1주차 기준) → 이상변동의 **72%를 놓침**. 경보 서비스 자체가 무의미
- 임계값 0.3으로 낮추면: "확률이 30%만 돼도 위험"이라 판정 → Recall 크게 상승
- **왜 0.1이나 0.2는 아닌가?** 너무 낮으면 경보 남발 → 사용자가 무시(양치기소년 효과)
- **왜 0.4·0.5가 아닌가?** 기본값 수준에선 Recall 개선이 미미
- **0.3은 Recall과 FP 폭주 사이의 휴리스틱 균형점**. 정식 최적값은 PR curve로 3주차에 튜닝 예정

### 30% 채택의 한계 (솔직 고백)
- 현재 0.3은 **데이터 기반 최적화 결과가 아니라 설계 의도 수치**
- 3주차에 `precision_recall_curve`로 각 임계값별 지표 뽑아서 데이터로 뒷받침 필요

## 5. 왜 매수추천 임계값 70%인가

### 한 줄 답
> **매수 추천이 틀리면 실제 금전 손실 → Precision 우선. 현재 매도 적중률 71.4%가 "70% 이상 달성 가능"을 시사해 심리적 기준선으로 설정**

### 논리
**① 비용 구조가 반대**
- 매수에서 FP(잘못된 매수) = 원금 손실
- 매수에서 FN(놓친 매수) = 기회비용 (원금은 유지)
- → **FP 비용 >> FN 비용** → **Precision 우선**

**② 1주차 실제 결과가 뒷받침**
- 1주차 기본 임계값에서 매수 적중률 33.3% → 100만원 시뮬레이션에서 **90만원으로 손실**
- 단순 보유(Buy & Hold)가 133만원인 것과 비교하면 **완패**
- "기본 설정 추천을 믿으면 돈을 잃음" → 임계값 올려야 함

**③ 왜 70%인가?**
- **매도 적중률 71.4%** 관측 → "확률 70% 이상일 때만 추천"이 현실적 달성 가능 수준
- 90% 이상으로 올리면 신호가 거의 안 나옴 → 서비스 사용성 0
- 70%는 "이 정도면 진짜 기회"라는 사용자 신뢰 심리적 기준선

## 6. 1주차 vs 2주차 전면 비교

### 구조 변화
| 항목 | 1주차 | 2주차 |
|------|-------|-------|
| **모델** | Random Forest 1개 | 5개 비교 (RF, LR, SVM, XGB, KNN) |
| **검증** | Random Split 80/20 | TimeSeriesSplit **5-Fold** |
| **정규화** | 미적용 | StandardScaler 적용 |
| **데이터 유출** | 있음 (미래가 학습에 섞임) | 차단 (시간 순서 보장) |
| **불균형 처리** | 미적용 | XGBoost `scale_pos_weight=1.8` |
| **최종 모델** | Random Forest | **XGBoost** 선정 |
| **서비스 설계** | 단일 모델 | 투트랙(위험경보 + 매수추천) |

### 성능 변화 (Recall 중심)
| 지표 | 1주차 (RF, Random Split) | 2주차 (XGB, TimeSeries 5-Fold) | 해석 |
|------|-------------------------|--------------------------------|------|
| Accuracy | 68.3% | 59.0% | **떨어짐** (↓) |
| Precision | 64.0% | 36.5% | **떨어짐** (↓) |
| **Recall** | 28.3% | **28.6%** | 비슷 (≈) |
| F1 | 39.3% | 32.4% | 떨어짐 (↓) |

### ⚠️ 중요: "성능 떨어진 거 아닌가?" 반박 논리
**수치는 내려갔지만 신뢰성은 올라갔다.**
- 1주차 Random Split은 **미래 데이터가 학습 셋에 섞임** → 실제 배포 시 달성 불가능한 부풀려진 점수
- 2주차 TimeSeriesSplit은 **현실적 예측 상황** 시뮬레이션 → 낮아 보여도 **실전에 가까운 값**
- "1주차 68% vs 2주차 59%" 비교는 **사과와 오렌지** 비교 — 평가 기준 자체가 달라짐
- **Recall은 오히려 미세 상승** (28.3% → 28.6%) → 본질적 성능은 유지 혹은 개선

## 7. 성능 지표 해석

### 4대 지표 (이진 분류)

**혼동행렬 (XGBoost 2주차 기준)**
|  | 예측: 정상 | 예측: 이상 |
|--|-----------|-----------|
| **실제: 정상** | TN 161 | FP 24 |
| **실제: 이상** | FN 48 | TP 26 |

| 지표 | 수식 | 값 | 해석 |
|------|------|------|------|
| **Accuracy** | (TP+TN)/전체 | 59.0% | 전체에서 맞춘 비율. **"정상"만 찍어도 64.5% 나오는 불균형 데이터라 믿으면 안 됨** |
| **Precision** | TP/(TP+FP) | 36.5% | 이상이라 예측한 것 중 진짜 이상 비율. "오경보 적게 울리고 싶을 때" 중시 |
| **Recall** | TP/(TP+FN) | **28.6%** | 실제 이상 중 감지한 비율. **"놓치고 싶지 않을 때" 중시 → 우리 목표** |
| **F1** | 2·P·R/(P+R) | 32.4% | Precision과 Recall의 조화평균. 균형 지표 |

### ROC·AUC
- ROC: TPR(Recall) vs FPR 곡선
- AUC: 곡선 아래 면적. 1에 가까울수록 좋음, 0.5면 무작위
- 2주차 XGBoost 기준 AUC는 별도 차트 참고 (`chart2_roc_slide.png`)

### 회귀 지표 (수익률 예측)
- **R²**: 모든 모델이 음수 → 평균 예측보다도 못함 → **회귀는 현재 실패 상태**
- 대응: 수익률 숫자는 **보조 지표**로만 취급, 분류 결과(이상변동 여부/방향) 중심으로 활용

## 8. 투트랙 서비스 설계 근거

### 왜 두 개의 서비스로 나눴나?
**교수님 피드백 반영 + 오류 비용 비대칭**

| | 위험 경보 🛡 | 매수 타이밍 🎯 |
|---|---|---|
| **틀렸을 때 피해** | 놓친 손실 (큰 FN 비용) | 잘못된 추천 = 실제 손실 (큰 FP 비용) |
| **우선 지표** | Recall | Precision |
| **임계값** | **0.30 (낮게)** | **0.70 (높게)** |
| **타겟 유저** | 변동성 걱정하는 개인투자자 | 신중한 기관투자자 |
| **UI 컨셉** | "놓치지 않기 우선" | "확실할 때만 추천" |
| **철학** | 의심스러우면 일단 알림 | 확실할 때만 추천 |

### B2C vs B2B 구분 (교수님 피드백)
- **B2C (개인)**: 위험경보 중심. UI/UX, 감정적 공감 메시지 중요
- **B2B (기관)**: 매수추천 중심. 데이터 품질, 일관성, 보고서화 용이성 중요
- 투트랙은 이 두 고객 니즈를 동시에 수용하는 구조

## 9. 교수님이 찌를 만한 허점 Q&A

> **원칙**: 방어는 짧게, 한계를 인정하고 3주차 개선으로 연결. 거짓말·회피 금지.

---

### Q1. "Recall 28%밖에 안 되는데 서비스라고 할 수 있나?"
**A.** 절대 수치는 낮지만 3가지 관점으로 봐야 합니다.
① 이는 **기본 임계값(0.5) 기준**이고, 투트랙 위험경보는 임계값을 0.3으로 낮춰 Recall을 의도적으로 높입니다.
② Recall 100%는 현실적으로 불가능하며, 저희 서비스는 **"놓치는 비율을 점진적으로 줄이는 것"**이 목표입니다.
③ 3주차에 **SMOTE, 시차 Feature, 하이퍼파라미터 튜닝**으로 0.4~0.5까지 개선 계획이 있습니다.

---

### Q2. "1주차보다 2주차 수치가 낮아졌는데 개선된 게 맞나?"
**A.** 수치는 낮지만 **신뢰성은 올라갔습니다**.
- 1주차 Random Split은 **미래 데이터가 학습 셋에 포함**된 잘못된 평가입니다 (Data Leakage).
- 2주차 TimeSeriesSplit은 **실제 배포 환경과 동일한 시간순 평가**입니다.
- 1주차 68%는 **실전에선 나올 수 없는 부풀려진 수치**이고, 2주차 59%가 **현실적 성능**입니다.
- **Recall은 미세 상승** (28.3 → 28.6)했고, 이게 본질적 개선의 증거입니다.

---

### Q3. "임계값 0.3은 왜 0.3인가? 데이터로 증명할 수 있나?"
**A.** 솔직히 말씀드리면 **현재 0.3은 설계 의도 기반이지 데이터 기반 최적값이 아닙니다**.
- 이유: ① 기본값 0.5에선 Recall 28%로 부족 ② 너무 낮추면 양치기소년 효과
- 3주차에 `precision_recall_curve`로 각 임계값별 Precision/Recall/F1을 산출해 **데이터 기반으로 재조정**할 계획입니다.
- 최종 값은 학계 표준이 아니라 우리 도메인(손실회피) 기준으로 정당화하겠습니다.

---

### Q4. "회귀 R²가 음수면 수익률 예측은 실패 아닌가?"
**A.** **맞습니다. 현재는 실패 상태입니다.**
- 원인: Feature가 당일 리스크 지표 7개뿐, 시계열 컨텍스트 부족
- 대응: ① UI에서 수익률 숫자를 **보조 지표**로만 표시 (분류 결과 중심) ② 3주차에 이동평균/RSI/볼린저밴드/시차 Feature 추가로 회귀 성능 개선 시도

---

### Q5. "GPR 지수는 뉴스 빈도 기반인데 실시간성이 있나?"
**A.** GPR은 두 버전이 있습니다.
- **Historical GPR** (월별, 1900~): 장기 추세용
- **Daily GPR** (일별, 1985~): **저희가 사용하는 버전**
- Daily GPR은 **매 거래일 갱신**되며, 저희 데이터도 주말·공휴일 제외 모든 평일에 다른 값이 들어갑니다.
- 다만 **공개 시점에 며칠 시차**가 있을 수 있어, 데모앱에서는 "위기지수 반영일"과 "시장지표 반영일"을 UI에 분리 표시해 투명성을 확보했습니다.
- 한계: 뉴스 기사 빈도 기반 지표라 **미리 예측은 불가**하고 사건 발생 후 반영되는 사후 지표 성격이 있음 → 보완책으로 4주차에 뉴스 감성 분석(NLP) 실험 예정

---

### Q6. "이상변동을 ±2%로 고정한 건 자의적이지 않나?"
**A.** **맞습니다. 현재 가장 큰 한계 중 하나입니다.**
- 변동성이 큰 시기(코로나, 전쟁 초기)엔 2%가 일상, 잠잠한 시기엔 2%가 대사건
- 개선안: **조건부 이상치 기준**(rolling 20일 표준편차의 2σ 등) → 시장 상황에 adaptive
- 3주차 후반 또는 제출④에서 반영 예정

---

### Q7. "왜 XGBoost인가? LightGBM, CatBoost은 비교 안 했나?"
**A.** 시간 제약으로 Scikit-learn 기본 5종 + XGBoost로 한정했습니다.
- XGBoost 선정 근거는 **Recall 1위, F1 1위, scale_pos_weight 내장** 3가지입니다.
- LightGBM/CatBoost는 **3주차 확장 비교**에 포함할 예정이며, 성능 개선 여지가 있다면 교체 검토합니다.

---

### Q8. "클래스 불균형이 35% vs 65%면 심한 건 아니지 않나?"
**A.** 일반적으로 **20% 이하를 심한 불균형**으로 봅니다. 맞는 지적입니다.
- 그럼에도 이 정도 불균형도 기본 모델 Recall을 떨어뜨리는 요인이 됩니다.
- 대응: ① `scale_pos_weight=1.8`로 가중치 보정 ② 3주차 SMOTE 오버샘플링으로 추가 개선 ③ 임계값 하향으로 Recall 우선 설계

---

### Q9. "하이퍼파라미터를 기본값으로 썼는데, 튜닝 안 한 결과인가?"
**A.** **네, 현재는 튜닝 전 상태입니다**. 의도적으로 남겨뒀습니다.
- 이유: 2주차 목표가 "모델 간 공정 비교"라서 **전부 기본값으로 출발선 정렬**이 우선
- 3주차에 GridSearchCV / RandomizedSearchCV로 XGBoost 전용 튜닝 → Recall, F1 개선 예상

---

### Q10. "실제 투자자가 이 서비스 믿고 돈 넣어도 되나?"
**A.** **아닙니다.** 데모앱 하단에도 명시했듯 **투자 권유가 아닌 참고용**입니다.
- 현재 수준은 학부 프로젝트 수준의 POC(개념 증명)입니다.
- 실서비스화 전 필요한 것: ① Recall 60%+ 수준 개선 ② 백테스팅 수년치 검증 ③ 규제 대응(금융위 신고 등) ④ 금융 도메인 전문가 검증

---

### Q11. "투트랙 서비스인데 두 모델이 같은 XGBoost이면 투트랙이 맞나?"
**A.** 날카로운 지적입니다. **모델은 같지만 의사결정 기준(임계값·행동 규칙)이 다릅니다**.
- 위험경보: 임계값 0.3 + 방향/수익률은 참고만
- 매수추천: 임계값 0.7 + 방향·수익률 일치 조건 + 변동성 낮을 때만
- 즉 **동일 모델, 다른 운영 정책(policy)** 구조입니다. 모델을 완전히 분리하는 건 3주차 이후 검토 과제입니다.

---

### Q12. "5-Fold 교차검증의 Fold 간 성능 편차는 어떤가?"
**A.** `chart4_fold_f1.png`에서 확인하실 수 있듯 XGBoost가 Fold 전반에 걸쳐 가장 **안정적**입니다.
- 일부 Fold(시장 급변동기)에선 성능이 떨어지지만, XGBoost 편차가 다른 모델보다 작습니다.
- 이게 XGBoost를 선정한 숨은 이유 중 하나입니다 (평균만 좋은 게 아니라 **분산도 낮음**).

## 10. 구술 대응 팁

### 발표 시 강조할 키워드
1. **"도메인 중심"** — 단순 수치 경쟁이 아닌 **손실회피** 심리·오류 비용 비대칭에 기반한 설계
2. **"신뢰성 있는 평가"** — TimeSeriesSplit으로 Data Leakage 차단
3. **"투트랙"** — 하나의 모델을 두 개의 정책으로 운영하는 구조적 차별화
4. **"점진적 개선"** — 1→2→3→4주차 각 단계별 명확한 개선 지점

### 방어 자세
- **한계는 먼저 인정** — 교수님 질문 나오기 전에 "현재 한계" 슬라이드에서 선제 고백
- **수치 하락은 구조 개선의 대가** — 1주차→2주차 Accuracy 하락을 공격받아도 Data Leakage 제거 설명
- **"3주차에 개선합니다"** 카드는 남발하지 말고, 실제 계획이 구체적인 항목에만 사용

### 피해야 할 답변
- ❌ "이건 학부 수준이라 한계가…" → 변명처럼 들림
- ❌ "XGBoost가 원래 좋아요" → 근거 없음
- ❌ "Accuracy가 높으니 괜찮아요" → 불균형 데이터에서 틀린 논리

### 추천 답변 형태
- ✅ "현재는 X이지만, 그 이유는 Y이고, Z로 개선 예정입니다"
- ✅ "지표 자체는 낮지만, **어떤 오류를 줄여야 하는가** 관점에서 Recall 우선 설계입니다"
